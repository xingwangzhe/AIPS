## 1 概述

### 1.1 设计背景

本文档是线上购药系统（OPS）的系统功能模块划分说明书。在前期需求分析阶段，我们已完成了43个功能需求点的定义，分布在药品浏览搜索、智能药师咨询、处方管理、购物车与订单、个性化服务、用户管理与健康档案六大业务域中。

系统功能模块划分是系统架构设计的关键步骤，其核心任务是将需求阶段定义的功能点按照高内聚、低耦合的原则组织为合理的功能模块，并提取公共的、可复用的基础服务模块。科学的模块划分能够降低系统复杂度，提高开发效率，保障系统的可维护性和可扩展性。

### 1.2 设计原则

系统功能模块划分遵循以下设计原则：

- **高内聚原则**：每个模块内部的功能紧密相关，共同服务于一个明确的业务目标。
- **低耦合原则**：模块之间通过清晰的接口进行交互，减少相互依赖。
- **单一职责原则**：每个模块只负责一类功能，避免职责混杂。
- **复用优先原则**：识别多个模块共同依赖的公共功能，提取为独立的基础服务模块。
- **分层解耦原则**：按照表现层、业务层、数据层进行分层，层与层之间通过标准接口通信。

## 2 功能模块总体架构

### 2.1 系统分层架构

线上购药系统采用经典的分层架构设计，自上而下划分为五层：表现层、网关层、业务服务层、公共基础服务层、数据层。各层职责清晰，层间通过标准接口通信，确保系统的松耦合和可扩展性。

**表 2-1 系统分层架构说明**

| 层次 | 职责 | 包含模块 |
|------|------|----------|
| 表现层 | 用户交互界面展示 | 移动APP、Web管理后台 |
| 网关层 | 统一入口、路由转发、鉴权限流 | API网关 |
| 业务服务层 | 核心业务逻辑处理 | 6个核心业务服务 |
| 公共基础服务层 | 公共功能复用服务 | 认证中心、消息中心、文件中心、支付中心 |
| 数据层 | 数据持久化与检索 | 关系数据库、缓存、搜索引擎、对象存储、AI模型 |

## 3 功能模块详细划分

### 3.1 表现层

表现层是用户与系统交互的界面，负责信息展示和用户操作接收。本系统包含两个前端应用：

**F-APP 移动APP（用户端）**
面向消费者的移动端应用，支持iOS和Android平台。承载所有C端业务功能，包括药品浏览、AI咨询、处方上传、下单支付、个人中心等。采用React Native跨平台框架开发。

**F-WEB Web管理后台（运营端）**
面向运营管理员和执业药师的Web端管理后台。提供商品管理、订单管理、处方审核、用户管理、数据统计等功能。采用React前端框架开发。

### 3.2 网关层

**G-GW API网关**
作为系统的统一入口，负责请求路由转发、身份认证鉴权、流量控制限流、请求日志记录、负载均衡等功能。所有前端请求必须通过网关访问后端服务。基于Spring Cloud Gateway或Nginx实现。

### 3.3 业务服务层

业务服务层是系统的核心，包含5个独立的业务服务，每个服务对应需求分析阶段定义的一个业务域。各服务之间通过API网关进行通信，避免直接耦合。

**S-DRUG 药品服务（Drug Service）**
负责药品信息的展示和检索。包含功能点：药品多级分类展示（FR-M01-01）、智能搜索（FR-M01-02）、搜索结果筛选与排序（FR-M01-03）、药品详情信息展示（FR-M01-04）、药品对比功能（FR-M01-05）、浏览历史记录（FR-M01-06）、热门搜索与搜索历史（FR-M01-07）、库存与配送信息实时展示（FR-M01-08）。

**S-CONSULT 智能咨询服务（Consult Service）**
负责AI药师咨询和症状自查功能。包含功能点：AI药师多轮对话咨询（FR-M02-01）、症状自查与导诊（FR-M02-02）、回答风险分级处理（FR-M02-03）、人工药师无缝转接（FR-M02-04）、咨询历史查询与管理（FR-M02-05）、药学知识库RAG检索增强（FR-M02-06）、咨询满意度评价（FR-M02-07）。

**S-PRESC 处方服务（Prescription Service）**
负责处方上传、识别和审核的全流程管理。包含功能点：处方拍照上传（FR-M03-01）、处方OCR识别与信息提取（FR-M03-02）、AI配伍禁忌与剂量检查（FR-M03-03）、药师人工审核确认（FR-M03-04）、处方审核状态跟踪（FR-M03-05）、处方档案管理（FR-M03-06）、电子处方对接预留（FR-M03-07）。

**S-ORDER 订单服务（Order Service）**
负责购物车管理、订单处理和支付履约。包含功能点：购物车商品管理（FR-M04-01）、订单确认与信息核对（FR-M04-02）、多渠道支付（FR-M04-03）、订单状态全程跟踪（FR-M04-04）、订单取消与退款（FR-M04-05）、隐私包装配送（FR-M04-06）、发票管理（FR-M04-07）、优惠券与促销活动（FR-M04-08）、售后服务入口（FR-M04-09）。

**S-USER 用户服务（User Service）**
负责用户管理、健康档案和个性化服务。包含功能点：用户注册与登录（FR-M06-01）、实名认证与身份核验（FR-M06-02）、收货地址管理（FR-M06-03）、健康数据采集与记录（FR-M06-04）、消息通知中心（FR-M06-05）、隐私设置与数据授权管理（FR-M06-06）、智能用药提醒（FR-M05-01）、个性化药品推荐（FR-M05-02）、复购提醒与一键续方（FR-M05-03）、健康档案管理（FR-M05-04）、健康知识内容推送（FR-M05-05）、家庭成员管理（FR-M05-06）。

### 3.4 公共基础服务层

公共基础服务层提取了各业务服务中重复出现的公共功能，形成独立的可复用服务。这些服务被多个业务服务共享调用，避免了功能的重复开发和维护。详细的合并说明见第4章。

**P-AUTH 认证中心（Auth Center）**
提供统一的用户身份认证和授权服务。被药品服务（浏览权限控制）、智能咨询服务（实名后才能咨询处方药问题）、处方服务（处方审核需验证用户身份）、订单服务（支付前身份验证）、用户服务（登录注册）共享调用。

**P-FILE 文件中心（File Center）**
提供统一的文件上传、存储和管理服务。主要被处方服务（处方图片上传）调用，同时支持药品服务（药品图片存储）和管理后台（运营素材上传）。

**P-PAY 支付中心（Payment Center）**
提供统一的支付渠道对接服务，封装微信支付、支付宝、银行卡等支付方式。主要被订单服务调用，通过标准化接口屏蔽不同支付渠道的差异。

**P-MSG 消息中心（Message Center）**
提供统一的消息推送服务，支持APP推送、短信、站内信三种渠道。被用户服务（用药提醒、系统通知）、订单服务（订单状态变更通知）、处方服务（审核结果通知）共享调用。

### 3.5 数据层

数据层为上层服务提供数据持久化、高速缓存、全文检索和文件存储能力。各类存储介质通过数据访问层封装，对业务服务透明。

**表 3-1 数据层组件说明**

| 组件 | 技术选型 | 用途 | 存储内容 |
|------|----------|------|----------|
| 关系数据库 | MySQL 8.0 | 业务数据持久化 | 用户、订单、处方、药品、咨询记录等结构化数据 |
| 缓存 | Redis 7.x | 热点数据缓存、会话管理 | 搜索热词、用户会话、库存状态等 |
| 搜索引擎 | Elasticsearch 8.x | 药品全文检索 | 药品名称、适应症、成分等可搜索字段 |
| 对象存储 | MinIO / OSS | 大文件存储 | 处方图片、药品图片、用户头像等 |
| AI模型服务 | PyTorch + Transformers | 大语言模型推理 | 药学知识库向量、LLM模型参数 |

## 4 重复模块合并说明

### 4.1 合并原则

在需求分析阶段定义的43个功能点中，部分功能存在跨模块的重复或相似性。在系统功能模块划分时，我们识别出以下类型的重复功能并进行合并：

- **认证类功能合并**：多个模块都需要用户身份验证，合并为统一的认证中心。
- **消息推送类功能合并**：多个模块都需要向用户发送通知，合并为统一的消息中心。
- **文件处理类功能合并**：多个模块都需要上传和存储文件，合并为统一的文件中心。
- **支付类功能合并**：订单模块中的支付功能抽象为独立的支付中心，为其他可能的付费场景预留扩展能力。

### 4.2 公共基础服务合并详情

**合并一：认证中心（P-AUTH）**

- 原分散功能：
  - 用户服务 - 用户注册与登录（FR-M06-01）
  - 用户服务 - 实名认证与身份核验（FR-M06-02）
  - 处方服务 - 处方审核需验证用户身份（FR-M03-04前置条件）
  - 订单服务 - 支付前身份校验（FR-M04-03）
- 合并理由：四个业务模块都需要用户身份认证能力，分散实现会导致认证逻辑重复、安全策略不统一。合并为认证中心后，所有服务的认证请求统一处理，确保安全策略的一致性。
- 合并效果：消除4处重复认证逻辑，统一安全标准，降低维护成本。

**合并二：消息中心（P-MSG）**

- 原分散功能：
  - 用户服务 - 消息通知中心（FR-M06-05）
  - 用户服务 - 智能用药提醒（FR-M05-01）
  - 订单服务 - 订单状态推送通知（FR-M04-04）
  - 处方服务 - 审核结果通知（FR-M03-05）
- 合并理由：四个模块都需要通过APP推送、短信、站内信等渠道向用户发送消息。合并为消息中心后，渠道对接、消息模板、发送策略统一管理，避免各服务重复对接消息渠道。
- 合并效果：统一3种消息渠道对接，集中管理消息模板和发送策略，支持消息频次控制和防骚扰机制。

**合并三：文件中心（P-FILE）**

- 原分散功能：
  - 处方服务 - 处方拍照上传（FR-M03-01）
  - 药品服务 - 药品图片展示（FR-M01-04）
  - 用户服务 - 用户头像上传
- 合并理由：文件上传、存储、压缩、格式转换等操作具有共性。合并为文件中心后，统一处理文件上传接口、图片压缩、存储路径管理、CDN分发等逻辑。
- 合并效果：统一文件上传接口规范，集中管理存储空间，支持图片自动压缩和水印。

**合并四：支付中心（P-PAY）**

- 原分散功能：
  - 订单服务 - 多渠道支付（FR-M04-03）
- 合并理由：虽然当前支付功能主要被订单服务使用，但未来可能扩展出会员充值、知识付费等场景。将支付能力抽象为独立服务，可复用性更强，也更便于对接新的支付渠道。
- 合并效果：封装支付渠道差异，统一支付接口和回调处理，便于新增支付方式的扩展。

**表 4-1 重复功能合并汇总**

| 公共基础服务 | 合并前分散数 | 被合并的业务服务 | 效果 |
|-------------|-------------|-----------------|------|
| 认证中心 P-AUTH | 4处 | 药品、咨询、处方、订单、用户 | 统一安全标准 |
| 消息中心 P-MSG | 4处 | 用户、订单、处方 | 统一消息渠道 |
| 文件中心 P-FILE | 3处 | 处方、药品、用户 | 统一文件管理 |
| 支付中心 P-PAY | 1处（预留扩展） | 订单 | 统一支付接口 |

## 5 模块间依赖关系

各功能模块之间存在依赖关系，主要体现在业务服务对公共基础服务的调用上。以下说明核心业务服务与公共基础服务之间的依赖关系：

**表 5-1 模块依赖关系矩阵**

| 业务服务 \ 公共服务 | 认证中心 P-AUTH | 消息中心 P-MSG | 文件中心 P-FILE | 支付中心 P-PAY |
|---------------------|-----------------|----------------|-----------------|----------------|
| 药品服务 S-DRUG | 需要（浏览鉴权） | 不需要 | 需要（药品图片） | 不需要 |
| 智能服务 S-CONSULT | 需要（用户鉴权） | 不需要 | 不需要 | 不需要 |
| 处方服务 S-PRESC | 需要（用户鉴权） | 需要（审核通知） | 需要（处方图片） | 不需要 |
| 订单服务 S-ORDER | 需要（支付鉴权） | 需要（状态通知） | 不需要 | 需要（支付处理） |
| 用户服务 S-USER | 需要（登录认证） | 需要（用药提醒） | 需要（头像上传） | 不需要 |

从依赖矩阵可以看出，认证中心被所有5个业务服务依赖，是系统中最基础的公共服务；消息中心被3个业务服务依赖；文件中心和支付中心的依赖范围相对集中。这种依赖分布合理，符合基础服务的定位。

### 6.2 模型反馈与应用

> **大模型反馈**

模型输出了完整的五层架构划分方案：

- **【分层架构建议】** - 表现层：移动APP + Web管理后台 - 网关层：API网关（统一入口、路由、鉴权、限流） - 业务服务层：6个微服务对应6大业务域 - 公共基础服务层：认证中心、消息中心、文件中心、支付中心 - 数据层：MySQL + Redis + ES + OSS + AI模型服务
- **【公共基础服务识别】** 模型识别出4个可合并的公共基础服务：
  1. 认证中心：合并用户认证、实名认证、身份核验（原分布在M02/M03/M04/M06）
  2. 消息中心：合并用药提醒、订单通知、审核通知（原分布在M03/M04/M05/M06）
  3. 文件中心：合并处方上传、药品图片、头像存储（原分布在M01/M03/M06）
  4. 支付中心：抽象多渠道支付，预留扩展能力
- **【模块依赖分析】** 模型分析了业务服务与公共服务的依赖矩阵，指出认证中心是唯一被所有业务服务依赖的模块，应作为最优先建设的基础服务。
- **【设计建议】** - 建议采用微服务架构，各服务独立部署 - 建议认证中心使用JWT令牌机制 - 建议消息中心支持消息模板和频次控制 - 建议文件中心支持图片自动压缩和CDN分发

模型输出的模块划分方案与人工分析结果高度一致。我们在模型建议的基础上进行了以下调整和确认：

- 采纳了模型提出的4个公共基础服务划分方案，与项目组独立分析结果完全一致。
- 采纳了模型建议的五层分层架构，作为系统整体架构设计的基础。
- 对模型建议的"6个业务服务"进行了微调，将M05个性化服务和M06用户管理合并为用户服务，因为两者高度相关且数据耦合紧密。

最终形成5个核心业务服务+4个公共基础服务的模块划分方案。
